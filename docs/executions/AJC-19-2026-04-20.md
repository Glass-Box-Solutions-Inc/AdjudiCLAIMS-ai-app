## TL;DR

Built a per-user training sandbox so new examiners can practice the full
AdjudiCLAIMS workflow against 9 hand-authored synthetic claims without ever
touching real PHI/PII. The toggle is mirrored from `User.trainingModeEnabled`
into the session at login, surfaces as a persistent yellow "TRAINING SANDBOX"
banner across the app shell, and exposes Enable / Disable / Reset / Status
endpoints under `/api/training/sandbox/*`. Synthetic claims are tagged
(`isSynthetic = true`) and owned (`syntheticOwnerId`) so multiple trainees in
one organisation cannot collide and so the reset operation can never reach
real claim data.

UPL anchor: synthetic claim data is GREEN-zone factual content. The same UPL
classifier, system prompts, and output validator that protect production data
apply identically inside the sandbox â€” the only thing that changes is the
data origin, never the legal posture of the AI's outputs.

## What shipped

### Schema

Two columns on `users`, two columns + two indexes on `claims`:

```sql
ALTER TABLE "users" ADD COLUMN "training_mode_enabled" BOOLEAN NOT NULL DEFAULT false;
ALTER TABLE "claims"
  ADD COLUMN "is_synthetic"        BOOLEAN  NOT NULL DEFAULT false,
  ADD COLUMN "synthetic_owner_id"  TEXT;
ALTER TABLE "claims"
  ADD CONSTRAINT "claims_synthetic_owner_id_fkey"
  FOREIGN KEY ("synthetic_owner_id") REFERENCES "users"("id")
  ON DELETE SET NULL ON UPDATE CASCADE;
CREATE INDEX "idx_claims_synthetic_owner" ON "claims"("synthetic_owner_id");
CREATE INDEX "idx_claims_is_synthetic"    ON "claims"("is_synthetic");
```

Migration: `prisma/migrations/20260423045225_training_sandbox_synthetic_claims/migration.sql`.

### Service

`server/services/training-sandbox.service.ts`:

- `getTrainingSandboxStatus(userId)` â€” returns `{ trainingModeEnabled, syntheticClaimCount, totalSyntheticClaims }`.
- `enableTrainingMode(userId, organizationId)` â€” flips the user flag, idempotently seeds the 9-scenario synthetic set, and returns the count.
- `disableTrainingMode(userId)` â€” flips the flag off; does **not** delete claims (trainees can re-enter without losing case-state).
- `resetSandbox(userId, organizationId)` â€” wipes only the calling user's synthetic claims (verified by belt-and-suspenders WHERE clause: `id IN (...) AND isSynthetic = true AND syntheticOwnerId = userId`) and re-seeds.

Helpers `seedSyntheticClaimsForUser` and `removeSyntheticClaimsForUser` are private to the service.

**Multi-trainee safety.** Each trainee's claim numbers are composite:
`${template.claimNumber}-${userId.slice(-8)}` â€” so trainees A and B in the
same org each own their own `TRAIN-001-...`, `TRAIN-002-...`, etc., without
unique-constraint collisions.

### Routes

`server/routes/training-sandbox.ts` â€” four endpoints, all `requireAuth()`:

| Method | Path                                | Action                              |
|--------|-------------------------------------|-------------------------------------|
| GET    | `/api/training/sandbox/status`      | Read flag + claim counts            |
| POST   | `/api/training/sandbox/enable`      | Enable training mode + seed claims  |
| POST   | `/api/training/sandbox/disable`     | Disable training mode (no delete)   |
| POST   | `/api/training/sandbox/reset`       | Wipe + re-seed user's synthetic set |

Intentionally exempt from `requireTrainingComplete` â€” the sandbox **is** the
training ground; gating it behind training-complete would defeat the purpose.

Each mutation:

1. Calls the service.
2. Mirrors the new value onto `request.session.user.trainingModeEnabled` so
   the banner reflects state immediately on the next request.
3. Audit-logs via `logAuditEvent` with `eventType: 'SYSTEM_CONFIG_CHANGED'`
   and structured `eventData` (`action`, `syntheticClaimCount`).

## Decisions & deferrals

### Per-user, not org-wide

The repo already had an org-wide `SANDBOX_MODE=true` env-gated path
(`server/services/sandbox.service.ts`) used for demos. That path is
preserved and untouched. AJC-19's per-user mode is additive: it lets a
single examiner enter sandbox while their colleagues stay in production
on the same deployment, which is what onboarding actually needs.

### `disableTrainingMode` does not delete claims

Trainees often disable + re-enable training while learning. Deleting on
disable would force a re-seed (new IDs, new URLs in the trainee's
browser history) every time. Reset is a separate, explicit action.

### Belt-and-suspenders delete WHERE

The reset path first selects `{ syntheticOwnerId, isSynthetic: true }`
to get IDs, then issues a `deleteMany` whose WHERE clause re-asserts both
flags **on top of** `id IN (...)`. This is intentional: even if a future
race interleaves a row whose ownership was nulled between SELECT and
DELETE, the DELETE cannot reach a non-synthetic or unowned row. There is
no path by which `resetSandbox` can touch real claim data.

### Did not write Playwright e2e

Service-level + route-level coverage at 16 tests pins the security and
correctness invariants. A full e2e (login, click toggle, seed, observe
banner, observe claims appearing on dashboard, reset, observe
disappearance) would belong in the `tests/e2e/` Playwright suite alongside
`auth-security.spec.ts`. Logged for future work; not in scope here.

### Did not seed across organisations

Synthetic claims are seeded against the trainee's own `organizationId`.
A cross-org "demo organisation" pattern was considered and rejected â€”
keeping the org boundary identical to production preserves the realism of
the training experience (same RBAC, same org-scoped queries, same audit
context).

## Compliance notes

- **UPL.** Synthetic data is GREEN-zone factual content. The classifier,
  system prompts, and output validator operate identically against
  synthetic and real data. There is no "training mode" branch in any UPL
  enforcement path â€” the sandbox is a data-source change, not a policy
  change.
- **PHI / PII.** All synthetic claimants are fictional, names suffixed
  `(Training)`. No real PHI ever enters the sandbox seeding path.
- **Audit.** All four mutating actions are audit-logged with
  `SYSTEM_CONFIG_CHANGED` event type and structured `eventData`. Status
  reads are not audit-logged (read-only, high-frequency, no policy
  change).
- **Multi-tenancy.** Reset cannot escape the calling user's synthetic
  claims â€” verified at unit level (WHERE clause assertion) and at design
  level (composite claim numbers prevent cross-trainee collision).
- **Secrets.** None touched.
